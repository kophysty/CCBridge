# Architecture Decision Records — CCBridge

**Архитектурные решения проекта.** Immutable. Один файл = одно
решение. Объясняют **что и зачем** выбрано, без code-discipline.

> ADR ≠ Rulebook ≠ Discovery/logs. См.
> [R-008](../Rulebook/R-008-workflow-logs-vs-rulebook.md).

---

## Когда создавать ADR

- Архитектурное решение, которое **меняет шаблон** (а не один кусок кода)
- Решение, которое **переживёт несколько релизов** и должно объясняться
  при онбординге
- Когда выбрана одна альтернатива из нескольких **и важно зафиксировать
  отвергнутые** для будущей сессии

## Когда НЕ создавать ADR

- Правило кода / процесса (это Rulebook)
- Разовое тактическое решение (это `Discovery/logs/decisions.md`)
- Документация модуля (это в самом коде или `docs/`)

---

## Шаблон ADR

```markdown
# ADR-NNN — <Заголовок одной строкой>

**Статус:** Proposed / Accepted / Superseded by ADR-MMM / Deprecated
**Дата:** YYYY-MM-DD
**Авторы:** kophy

---

## Контекст

В каком положении был проект, какая возникла потребность.
Конкретно, без воды.

## Решение

Что выбрали.

## Альтернативы

Что рассматривали и почему отвергли. Эта секция важна — без неё
через год никто не поймёт «почему не X».

## Последствия

Положительные: …
Отрицательные / trade-offs: …
Что становится сложнее в результате этого решения.

## Связи

Rulebook R-NNN, другие ADR, ссылка на обсуждение в decisions.md.
```

---

## Реестр ADR

```
  ID       Дата          Статус       Заголовок
  ───────  ────────────  ──────────  ─────────────────────────────────────
  ADR-001  2026-04-28    Accepted    Python CLI вместо bash, Node.js или
                                      сразу MCP server
```

---

## Lifecycle

- **Proposed** — обсуждается, не применяется
- **Accepted** — действует
- **Superseded by ADR-MMM** — заменено новым решением, файл оставлен для
  истории
- **Deprecated** — больше не действует, не было заменено

ADR **никогда не удаляются**, даже Superseded. Это immutable history.
