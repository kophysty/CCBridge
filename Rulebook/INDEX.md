# Rulebook — CCBridge

**Реестр активных правил кода и процесса.** Атомарные файлы
`R-NNN-<slug>.md`, каждое правило — одна тема.

> **Перед кодом сверяться обязательно.** Это не справочник «на потом» —
> это рабочий чек-лист. Каждое правило накопилось из реальной боли
> (своей или заимствованной из Oil Automation).
> Процесс добавления — [R-000](R-000-how-to-add-rules.md).

---

## Приоритеты

```
  Уровень   Что значит
  ───────   ────────────────────────────────────────────────────────
  🔴 P0      Нарушение = data corruption / security hole / silent
             prod break. Защищается тремя слоями (code + CLAUDE.md
             + regression test). Не нарушать никогда.

  🟡 P1      Нарушение = срыв доверия пользователя / потеря работы.
             Защищается двумя слоями. Нарушать только с явной
             командой.

  🟢 P2      Нарушение = технический долг / снижение качества.
             Один слой защиты (CLAUDE.md). Можно отступать
             осознанно с обоснованием.

  ⚪ P3      Стилистика / preferences. Не блокирует, но желательно.
```

---

## Категории

```
  Тег              Что значит
  ───────────────  ────────────────────────────────────────────────
  antipattern      «Что не делать» (R-001 запрет коммитов без ОК)
  pattern          «Как делать правильно»
  workflow         Процесс (R-005 TDD, R-007 planning discipline)
  preference       Стилистика / соглашение (R-003 ASCII tables)
```

## Активные правила

```
  ID      Приоритет   Категория       Название                                  Защита
  ──────  ──────────  ──────────────  ────────────────────────────────────────  ──────────
  R-000   meta         workflow         Как добавлять правила                     —

  R-001   🟡 P1        workflow         Нет commit/push без явного approval        CLAUDE.md
  R-002   🟡 P1        workflow         Нет удаления debug-логов без ОК            CLAUDE.md
  R-003   ⚪ P3        preference       ASCII-таблицы в .md с выровненными         CLAUDE.md
                                         колонками
  R-004   ⚪ P3        preference       Версионный подход для файлов               CLAUDE.md
                                         (v1 → v2 → v3, не перезапись)
  R-005   🟢 P2        workflow         TDD по умолчанию для новых модулей         CLAUDE.md
  R-006   🟢 P2        workflow         verification-before-completion             CLAUDE.md
                                         (тесты + evidence перед «готово»)
  R-007   🟡 P1        workflow         Planning discipline — plan-файл +          CLAUDE.md
                                         запись в ROADMAP в одном коммите
  R-008   🟢 P2        workflow         Rulebook ≠ Discovery/logs                  CLAUDE.md
                                         (правила vs нарратив)
```

**Защита** — какие слои реализованы:
- **CLAUDE.md** — ссылка из главного файла
- **code** — helper / lint rule / pre-commit hook
- **test** — regression test, который ловит нарушение

## Обязательные сверки перед задачей

```
  Тип задачи                                  Сверить с
  ──────────────────────────────────────────  ───────────────────────
  Любые изменения в репо CCBridge             🟡 R-001 (commits)
                                                R-008 (logs vs rules)

  Новый модуль / тест                          🟢 R-005 (TDD)
                                                R-006 (verification)

  Удаление дебаг-логирования                   🟡 R-002

  План спринта / spike                          🟡 R-007 (запись в ROADMAP)

  Markdown с таблицами                          ⚪ R-003

  Создание новой версии файла                   ⚪ R-004 (не перезаписывать)

  Запись нового опыта (повторяющийся кейс)     R-000 (как добавить)
```

## Lifecycle

- **active** — применяется
- **deprecated** — устарело, заменено на `R-NNN-superseded-by-R-MMM.md`,
  файл оставляем для истории (не удаляем)

Новые правила добавляются по [R-000](R-000-how-to-add-rules.md).
