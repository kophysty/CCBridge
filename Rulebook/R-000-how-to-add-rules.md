# R-000 — Как добавлять правила в Rulebook

**Приоритет:** мета
**Категория:** workflow
**Статус:** active
**Введено:** 2026-05-02
**Источник:** заимствовано из Oil Automation `Rulebook/R-000-how-to-add-rules.md`

---

## Контекст

Когда мы наступаем на грабли (антипаттерн) или находим хороший приём
(паттерн), это нужно зафиксировать. Иначе через месяц повторим ту же
ошибку или не вспомним удачное решение.

Мы **не складываем** новые правила в `Discovery/logs/decisions.md`
как очередное ADR — там нарратив (как пришли к решению). Активные
правила живут в Rulebook как атомарные файлы. См. [R-008](R-008-workflow-logs-vs-rulebook.md).

---

## Правило

Каждое накопленное правило = **отдельный файл** `R-NNN-<slug>.md`
по фиксированной структуре. Реестр в [INDEX.md](INDEX.md).

### Когда добавлять правило

- Повторяющаяся ситуация (≥2 раза наступили на одни грабли)
- Заимствование из другого проекта, который уже эту боль пережил
- Архитектурное решение, которое требует **дисциплины при коде**
  (если решение разовое и не требует постоянной сверки → ADR, не Rulebook)

### Когда НЕ добавлять

- Разовый случай, не повторится (это просто `Discovery/logs/decisions.md`)
- Архитектурное решение без code-discipline (это ADR)
- Жалоба «мне не нравится» без аргумента (это preference в CLAUDE.md
  или P3 правило с обоснованием)

---

## Процесс (5 шагов)

### Шаг 1: выбрать ID

Следующий свободный номер `R-NNN`. Смотрим [INDEX.md](INDEX.md), берём
максимум + 1. Никогда не переиспользуем номера, даже от deprecated
правил.

### Шаг 2: создать файл

`Rulebook/R-NNN-<slug>.md` по шаблону:

```markdown
# R-NNN — <Заголовок одной строкой>

**Приоритет:** P0 / P1 / P2 / P3
**Категория:** antipattern / pattern / workflow / preference
**Статус:** active
**Введено:** YYYY-MM-DD
**Источник:** что породило правило (инцидент / заимствование / спайк)

---

## Контекст

Что случилось / какая ситуация порождает необходимость в правиле.
Конкретный пример из реальности.

---

## Правило

Чёткая формулировка одним абзацем. **Что делать / чего не делать.**

# Конкретный код, который ломает правило
```

# Конкретный код, который соблюдает правило
```

## Как проверить (защита)

- **CLAUDE.md** — ссылка из главного файла (минимум для P2-P3)
- **code** — helper / lint rule / pre-commit hook (для P1+)
- **test** — regression test (для P0)

Указать конкретные файлы / тесты, реализующие защиту.

---

## Связанные правила

R-NNN, R-MMM (если есть).
```

### Шаг 3: добавить в INDEX.md

Открыть [INDEX.md](INDEX.md), вставить строку в таблицу «Активные
правила» в правильном месте (по ID). Обновить таблицу
«Обязательные сверки» если применимо.

### Шаг 4: реализовать защитные слои

Для **P0**: обязательны все три слоя (CLAUDE.md ссылка + code helper
+ regression test). Без полного покрытия правило остаётся **paper rule**
и не работает.

Для **P1**: минимум CLAUDE.md + один из (code / test).

Для **P2-P3**: достаточно CLAUDE.md.

### Шаг 5: одиночный коммит

```
docs(rulebook): R-NNN — <короткое название>

- Создан Rulebook/R-NNN-<slug>.md
- INDEX обновлён
- CLAUDE.md ссылка добавлена
- (опционально) защитный код / тест
```

---

## Lifecycle

- **active** → правило применяется
- **deprecated** → правило устарело. **Не удаляем файл**, добавляем
  в шапку:
  ```
  **Статус:** deprecated, superseded by R-MMM
  ```
  В INDEX.md — переносим в раздел «Deprecated» (внизу).

---

## Antipattern: 152-й ADR в decisions.md

**Не делать:** копить новые правила в `Discovery/logs/decisions.md`
как 1-й, 2-й, … 152-й «архитектурный пункт». Логи — нарратив. Когда
их станет 200, ни одно «активное» правило никто не найдёт.

**Делать:** новые правила сразу в Rulebook как атомарный файл.
В decisions.md — ссылка на R-NNN.
