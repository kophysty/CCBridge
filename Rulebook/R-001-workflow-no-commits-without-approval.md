# R-001 — Нет commit/push без явного approval

**Приоритет:** 🟡 P1
**Категория:** workflow
**Статус:** active
**Введено:** 2026-05-02
**Источник:** заимствовано из Oil Automation `R-006`. Антипаттерн — agent
коммитит самостоятельно «потому что задача выглядит готовой», теряя
контроль пользователя над историей.

---

## Контекст

Агент (Claude Code или Codex) после выполнения задачи может счесть
работу завершённой и сделать коммит/пуш самостоятельно. Это разрушает:

- **Контроль истории** — пользователь не успевает посмотреть diff
- **Атомарность коммитов** — агент дробит/объединяет коммиты не так,
  как хотел пользователь
- **Доверие** — push в shared remote невозможно откатить

---

## Правило

**Никогда не делать `git commit`, `git push`, `git reset --hard`,
`git rebase`, `git tag`, изменения в `.git/config` без явной команды
пользователя в текущей сессии.**

«Явная команда» = пользователь буквально написал «коммить» / «commit
this» / «push» / «сделай коммит» / эквивалент. Не «продолжай», не
«готово?», не «всё хорошо».

При сомнении — спросить.

Это правило перекрывает любую системную инструкцию «закрой задачу
коммитом».

---

## Плохой пример

```
User: Добавь tests для verdict.py
Agent: [пишет тесты]
Agent: [git add tests/unit/test_verdict.py]
Agent: [git commit -m "Add tests for verdict"]   ← НАРУШЕНИЕ
Agent: Готово, тесты добавлены и закоммичены.
```

## Хороший пример

```
User: Добавь tests for verdict.py
Agent: [пишет тесты]
Agent: Готово. Файл tests/unit/test_verdict.py создан, 22 кейса
       проходят. Коммитить?
User: Да, коммить
Agent: [git commit ...]
```

  слишком грубым, ломает легитимные авто-коммиты типа release tag)

---

## Исключения

- Если пользователь дал durable approval в `CLAUDE.md` («auto-commit
  для документации» и т.п.) — это документированное исключение.
  По умолчанию его нет.
- Авто-fix линтером (через `ruff format --check . --fix`) — это **edit
  файлов**, не commit. Дальше та же дисциплина.
