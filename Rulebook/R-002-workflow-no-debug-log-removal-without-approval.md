# R-002 — Нет удаления debug-логов без согласования

**Приоритет:** 🟡 P1
**Категория:** workflow
**Статус:** active
**Введено:** 2026-05-02
**Источник:** заимствовано из Oil Automation. Антипаттерн — agent после
закрытия бага «убирает за собой» все `print` / `logger.debug` / временные
ассерты, теряя инструменты для следующего инцидента.

---

## Контекст

Когда баг закрыт, искушение «прибраться» — удалить дебаг-логи,
временные `print()`, ассерты, отладочные комментарии. Но:

- Следующий похожий баг будет дольше диагностировать
- Дебаг-логи могут защищать от регрессии (видимый сигнал в проде)
- Иногда они оставлены сознательно (TODO, временный workaround)

Решение об удалении — **за пользователем**, не за агентом.

---

## Правило

**Не удалять самостоятельно** ничего из перечисленного, даже если
кажется что задача «готова»:

- `print()` calls
- `logger.debug(...)` / `logger.info(...)`
- временные `assert` для отладки
- комментарии вида `# TODO`, `# FIXME`, `# DEBUG`, `# HACK`
- `breakpoint()` (это очевидно, но всё равно)

При закрытии задачи — **спросить пользователя**: «оставить debug-логи
или прибраться?». При неоднозначности — оставить.

---

## Плохой пример

```python
# До
def calculate_price(qty, rate):
    logger.debug(f"calc: qty={qty} rate={rate}")
    result = qty * rate
    logger.debug(f"result={result}")
    return result

# Agent после фикса бага:
def calculate_price(qty, rate):
    return qty * rate              # ← удалил debug-логи без спроса
```

## Хороший пример

```
Agent: Баг в calculate_price пофикшен (был int overflow при rate=0).
       В функции есть два logger.debug. Оставить или убрать?
User: Оставь, пригодятся
Agent: [не трогает]
```

## Исключения

- Удаление `breakpoint()` — допустимо без спроса (это очевидная
  забытая отладка, никогда не нужна в продакшене).
- Удаление debug-логов **в рамках одного PR**, который пользователь
  явно описал как «cleanup» — допустимо.
