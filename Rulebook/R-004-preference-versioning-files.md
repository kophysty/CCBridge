# R-004 — Версионный подход для файлов (v1 → v2 → v3)

**Приоритет:** ⚪ P3
**Категория:** preference
**Статус:** active
**Введено:** 2026-05-02
**Источник:** заимствовано из Oil Automation `R-015`. Перезапись оригинала
теряет историю и контекст принятия решений.

---

## Контекст

Когда документ (план, диаграмма, спецификация, ARCHITECTURE.md)
существенно меняется, есть искушение перезаписать оригинал «новой
улучшенной версией». Но:

- Теряется возможность сравнить `v1 vs v2` в обсуждении
- Git-история помогает, но имена файлов не отражают эволюцию
- При откате решения непонятно, к какой версии возвращаемся

Это касается **документации и артефактов**, не исходного кода
(в коде версионирование делает git).

---

## Правило

Для документов / диаграмм / спецификаций / планов:

- **Никогда не перезаписывать оригинал** значимыми изменениями
- Создавать новую версию: `architecture-plan-v1.md` → `v2.md` → `v3.md`
- В новой версии в шапке писать «Заменяет: v2 (см. для сравнения)»
- Старые версии **не удаляем** (как минимум на время текущей фазы
  проекта)

«Значимое изменение» = переписан один или несколько разделов; смена
архитектурного подхода; пересмотр scope.

«Незначимое изменение» (можно править оригинал) = опечатки,
форматирование, новые ссылки, добавление AC без изменения
существующих.

---

## Плохой пример

```
ARCHITECTURE.md (v1) ───[существенно переписали]───> ARCHITECTURE.md
                                                       ↑
                                                 v1 безвозвратно
                                                 потеряна (ну,
                                                 в git есть, но
                                                 без явных меток
                                                 версий)
```

## Хороший пример

```
ARCHITECTURE.md          ← живой документ, в шапке "версия v0.0.3"
ARCHITECTURE-v0.0.1.md   ← snapshot перед существенной правкой
ARCHITECTURE-v0.0.2.md   ← snapshot перед следующей
```

Или метаданные в шапке:

```markdown
# CCBridge — Architecture

**Версия:** v0.0.3-draft
**Заменяет:** v0.0.2 (audit-applied, см. git history at <hash>)
**Заменено в:** —
```

## Как проверить (защита)

- **CLAUDE.md** — раздел про версионирование документов
- **AGENTS.md** — раздел «Documentation Versioning»

---

## Исключения

- **Output/** (READ-ONLY) — финализированные deliverables никогда не
  редактируются, новая версия = новый файл. Это даже строже R-004.
- **Исходный код** (`.py`, `.toml`, etc.) — версионирование через git,
  не имена файлов.
