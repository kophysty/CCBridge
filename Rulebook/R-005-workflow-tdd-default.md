# R-005 — TDD по умолчанию для новых модулей

**Приоритет:** 🟢 P2
**Категория:** workflow
**Статус:** active
**Введено:** 2026-05-02
**Источник:** инженерная практика, синхронизировано с скиллом
`test-driven-development`. Без этого код «вроде работает», тесты
догоняют постфактум и часто поверхностные.

---

## Контекст

Когда тесты пишутся **после** имплементации, они склонны:

- Подтверждать то что код делает, а не то что должен делать
- Пропускать edge cases (не приходят в голову, когда код уже работает
  на golden path)
- Быть «галочкой», а не спецификацией поведения

TDD ставит тест в роль **исполняемой спецификации**. Сначала
формулируется ожидание, потом код.

---

## Правило

Для **новых модулей в `core/` и `transports/`** — порядок работы:

1. Написать failing test для одного аспекта поведения
2. Написать **минимальный код**, чтобы тест прошёл (не больше)
3. Refactor если нужно (тест защищает)
4. Следующий аспект → следующий test

**Не относится** к:
- Тривиальным модулям (`__init__.py`, экспорты)
- CLI entry points (тестируются e2e)
- Эксперименты / spike (явно помечены)

---

## Плохой пример

```python
# Шаг 1: написать сразу всё
class CCBridgeLock:
    def __init__(self, path): ...
    def acquire(self): ...
    def release(self): ...
    def is_stale(self): ...
    def takeover(self): ...

# Шаг 2: написать тесты после
# Шаг 3: тесты «подтверждают» что код делает то, что делает,
#         даже если поведение неправильное.
```

## Хороший пример

```python
# tests/unit/test_lockfile.py
def test_acquire_creates_lockfile():
    lock = CCBridgeLock(tmp_path / "lock")
    lock.acquire()
    assert (tmp_path / "lock").exists()

# Запуск → fail (CCBridgeLock не существует)

# src/ccbridge/core/lockfile.py — минимум, чтобы test прошёл
class CCBridgeLock:
    def __init__(self, path): self.path = path
    def acquire(self): self.path.touch()

# Запуск → pass

# Следующий test:
def test_release_removes_lockfile(): ...
# И так по одному аспекту.
```

---

## Исключения

- Если модуль маленький (≤30 строк, очевидная логика) — можно
  «характеризационным» тестом одной батчей.
- Если переписываем существующий модуль — golden tests **сначала**
  на текущее поведение, потом рефакторинг.
