# R-006 — verification-before-completion

**Приоритет:** 🟢 P2
**Категория:** workflow
**Статус:** active
**Введено:** 2026-05-02
**Источник:** скилл `verification-before-completion`. Без evidence
утверждение «готово / починено / работает» — ничем не подкреплено
и часто оказывается ложным.

---

## Контекст

Самый частый источник потерянного времени и доверия: агент говорит
«готово, тесты прошли», пользователь верит, через час приходит
фидбек «не работает». Причины:

- Тесты не запускались, агент предположил
- Тесты запускались на старом коде
- Тесты прошли, но не проверяют то, о чём заявил агент
- Запущена не та команда (например, `pytest tests/unit/test_X.py`
  вместо всех тестов, и регрессия в другом модуле осталась незамеченной)

Решение: **никаких success-claims без evidence**.

---

## Правило

Перед тем как сказать «готово / fixed / passed» — выполнить и показать:

1. **Что именно проверено** (конкретная команда: `pytest tests/unit/`,
   `ruff check src/`, ручной smoke по сценарию)
2. **Реальный вывод** (последние 10-30 строк stdout, не парафраз)
3. **Соотношение к утверждению**: «`pytest` прошёл 22 теста — это
   подтверждает работу `verdict.py`, но не проверяет интеграцию с
   orchestrator (туда тестов пока нет)»

Если вывод не показан — declaring done запрещено.

---

## Плохой пример

```
Agent: [пишет код]
Agent: Готово, тесты проходят.   ← где evidence?
```

## Хороший пример

```
Agent: [пишет код]
Agent: [запускает pytest]
Agent: pytest tests/unit/test_verdict.py
       =========================== test session starts ==========================
       collected 22 items
       tests/unit/test_verdict.py ......................     [100%]
       =========================== 22 passed in 0.34s ===========================

       Готово. 22/22 кейсов test_verdict.py проходят. Это покрывает
       Pydantic validation и semantic validation в изоляции; интеграция
       с orchestrator не проверена (orchestrator ещё не написан).
```

## Исключения

- При работе в режиме `--print` / non-interactive (например, hook),
  где stdout фиксированно структурирован — evidence в виде exit code +
  count.
- Тривиальные правки (опечатка, форматирование) — можно без полного
  pytest, но опять же показав что именно сделано (diff).
