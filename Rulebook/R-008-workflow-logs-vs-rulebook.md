# R-008 — Rulebook ≠ Discovery/logs (правила vs нарратив)

**Приоритет:** 🟢 P2
**Категория:** workflow
**Статус:** active
**Введено:** 2026-05-02
**Источник:** заимствовано из Oil Automation `R-012`. Без чёткого
разделения новые правила оседают в `decisions.md` 152-м пунктом и не
работают как чек-лист.

---

## Контекст

В проекте копится два разных типа знания:

- **Активные правила** — «что делать в коде / процессе». Читаются перед
  каждой задачей. Должны быть короткими, атомарными, грепаемыми.
- **Нарратив** — «как пришли к решению, какие были альтернативы, что
  обсуждали с пользователем». Полезен для истории и контекста, но не
  для повседневной работы.

Если их смешать (всё в `decisions.md` или всё в `Rulebook/`):

- Правила тонут в нарративе → перестают применяться
- Нарратив теряется в правилах → новые сессии не понимают почему
  было принято решение

---

## Правило

**Rulebook/** — атомарные **активные правила** (`R-NNN-<slug>.md`).
Читаем перед кодом. Каждое правило — одна тема, фиксированная
структура.

**Discovery/logs/** — нарратив:
- `decisions.md` — append-only, формат Was/Now/Why/Impact
- `insights.md` — наблюдения с источником
- `conversation-log.md` — переписка / сессии
- `*-checkpoint-*.md` — handoff между агентами
- `*-spike-*.md`, `*-postmortem-*.md`, `*-audit-*.md` — отдельные файлы
  на крупные события

**ADR/** — архитектурные решения (immutable, что-зачем). Между Rulebook
и decisions: правило кода → Rulebook, разовое архитектурное решение → ADR,
обсуждение как пришли к ADR → decisions.md.

---

## Плохой пример

`Discovery/logs/decisions.md`:

```markdown
### [2026-04-15] Использовать tabs only в .md
... обсуждение ...

### [2026-04-16] Mock'ать БД нельзя в integration tests
... обсуждение ...

### [2026-04-17] TDD для новых модулей
... обсуждение ...

### [2026-04-18] ... ещё 148 пунктов ...
```

→ Никто из этих 152 «решений» не читается перед задачей. Правила
de facto не существуют.

## Хороший пример

`Rulebook/R-005-workflow-tdd-default.md`:

```markdown
# R-005 — TDD по умолчанию для новых модулей
[чёткое правило, плохой/хороший пример, защита]
```

`Discovery/logs/decisions.md`:

```markdown
### [2026-05-02] Принят TDD как default для core/

**Was:** не было дисциплины, тесты появлялись постфактум.

**Now:** TDD обязателен для модулей в `core/` и `transports/`,
формализовано как [R-005](../../Rulebook/R-005-workflow-tdd-default.md).

**Why:** характеризационные тесты постфактум подтверждают то что код
делает, не то что должен делать. (Ссылка на инцидент / обсуждение).

**Impact:** новые PR проходят review с проверкой test-first порядка.
```

## Как проверить (защита)

- **CLAUDE.md** — раздел «Разделение источников знания»
- **AGENTS.md** — раздел «Documentation Hierarchy»

## Decision tree

```
  Хочешь зафиксировать что-то...

  Это активное правило кода/процесса,
  которое должно применяться повседневно?
   ├─ Да   → Rulebook/R-NNN-<slug>.md (через R-000)
   └─ Нет  → ↓

  Это разовое архитектурное решение
  (что и зачем выбрали, immutable)?
   ├─ Да   → ADR/ADR-NNN-<slug>.md
   └─ Нет  → ↓

  Это нарратив (как обсуждали, какие были
  альтернативы, переписка с клиентом)?
   ├─ Да   → Discovery/logs/decisions.md или insights.md
   └─ Нет  → возможно вообще не нужно фиксировать
```
